Smooth Snake — Architecture Plan

A frontend-only browser game · vanilla JS + Canvas · zero dependencies · ~400–700 lines

Goal. A snake game with smooth, continuous movement: the snake steers toward the mouse cursor, gets longer and faster with every pickup, and dies on spike hazards scattered randomly across the arena — placed so that every part of the arena always stays reachable. Nothing exists yet; this plan is the blueprint for a fresh smooth-snake/ folder.

1 · Module structure & data flow

Pure static files, ES modules, no build step. No event bus and no class hierarchies — a single plain game state object is created in main.js and passed by reference into update(game, dt) and render(game, ctx) functions exported by focused modules.

10 modules, each 30–90 lines.

Game loop & states

requestAnimationFrame with delta-time, clamped to MAX_DT = 1/30 s so a backgrounded tab can't teleport the head across the arena. No fixed timestep — nothing here needs determinism.

State is one string: 'menu' → 'playing' → 'gameover' (click/Enter to start or restart). update() early-returns unless playing; render() always draws the arena and overlays the current screen.

2 · The four algorithms that matter

Mouse steering — clamped angle turn

The head has position and heading θ. Each frame: compute the angle to the cursor, wrap the difference to [−π, π], then clamp the turn to TURN_RATE · dt. The clamp is the handling model — the snake can't turn on a dime.

d = atan2(sin(t−θ), cos(t−θ))
θ += clamp(d, −TURN_RATE·dt, +TURN_RATE·dt)
x += cos(θ)·speed·dt
y += sin(θ)·speed·dt

Smooth body — arc-length trail

Record the head's path densely every frame. Place body segments along it at fixed arc-length spacing (interpolating between recorded points), so segments stay evenly spaced at any speed. Growth is just segmentCount += SEGMENTS_PER_FOOD — the tail extends naturally as the path trim length grows. Trim the path past segmentCount · SEGMENT_SPACING + margin.

Speed ramp — linear with cap

speed = min(BASE_SPEED
    + eaten · SPEED_PER_FOOD,
      MAX_SPEED)

Past the cap, difficulty keeps escalating anyway: your own length crowds the arena. Optional tuning hook (off by default): reduce TURN_RATE slightly as speed rises.

Spike placement — spacing + flood-fill

Layer 1 — dart throwing: accept a random spike only if it's at least 2·spikeR + CORRIDOR_WIDTH from every other spike (corridor ≈ 1.6× body width), and clear of walls, spawn, and food. The spacing rule directly guarantees threadable gaps.

Layer 2 — validation: overlay a coarse grid, mark cells near spikes blocked, BFS from spawn. Any unreachable free cell → regenerate the whole layout. Layer 1 makes this almost always pass; layer 2 makes it a guarantee, in ~25 lines.

Collision & death rules

Everything is circle-vs-circle against the head — cheap, and tunneling-safe because dt is clamped. Head vs food → eat, grow, speed up, respawn food clear of hazards. Head vs spike or wall → death. Head vs own body → death, but the first NECK_GRACE_SEGMENTS (~4) behind the head are skipped so the neck can't self-kill.

3 · Rendering

Fixed logical arena (900 × 600) scaled and centered to the window, so spike math never depends on screen size. Canvas backing store is sized at devicePixelRatio with one ctx.scale(dpr, dpr) per resize. Draw order: background → food (pulsing circle) → spikes (radial spike shapes, warm color) → snake (round-capped stroked path through segment centers, distinct head with eyes oriented by heading) → HUD (fillText score / high score) → menu or game-over overlay. High score persists in localStorage.

4 · Implementation order

Eight milestones, each independently runnable:

  1. Skeleton: canvas, DPR/arena scaling, delta-time RAF loop, moving test dot. Testable: dot animates.
  2. Steering: mouse tracking, clamped-turn head movement, walls kill. Testable: drive the head.
  3. Body trail: path recording + arc-length sampling, stroked-path render. Testable: smooth trailing snake.
  4. Food + growth + speed ramp + HUD score. Testable: the core loop — eat, grow, speed up.
  5. Spikes: dart-throwing placement, flood-fill validation, spike death. Testable: hazards kill, no unreachable pockets.
  6. Self-collision with neck grace; head/eyes; respawn clearance; visuals.
  7. State machine, menu & game-over overlays, localStorage high score.
  8. Tuning pass on config.js; optional body taper.

6 · Risks & failure modes

Tunneling at max speed. At 340 px/s and 30 fps the head moves ~11 px/frame, below the smallest collision sum (head 10 + spike 12 = 22 px), so the dt clamp alone prevents pass-through. If constants are ever tuned hotter, add head-movement sub-stepping.
Over-packed spike config. Dart throwing has an attempt budget (2000); if tuning ever makes SPIKE_COUNT unsatisfiable, generation degrades gracefully to fewer spikes rather than looping forever.
Mouse outside the canvas. The head keeps steering toward the last known cursor position — no special case needed, but worth feeling out during the tuning pass.

7 · Decisions — resolved

All three open questions were answered on 2026-07-04 and are baked into the plan above.

Wall behavior → walls kill ✓

Consistent with the spike-hazard theme; steering skill matters near edges. Wrap-around remains a possible future config swap.

How the game is run → static server + ES modules ✓

Run with python -m http.server (or any static server) from smooth-snake/; the clean 10-module split stays. Chrome blocks module imports on file://, so double-clicking index.html is not supported.

Arena sizing → fixed logical arena, scaled to fit ✓

All game math runs in a resolution-independent 900 × 600 arena, scaled and centered to the window; spike layout stays valid on any screen or resize.
